MLOps Learning Project

This project was built while learning MLOps practices, specifically to get hands-on experience with experiment tracking, model versioning, containerization and CI/CD deployment pipelines in a real deep learning context.

This project combines deep learning for medical image classification with a full MLOps pipeline, the goal being not just to build a working classifier but to practice the infrastructure and workflows that make ML models reproducible, trackable and deployable in production.

The classification task involves detecting chest cancer from CT scan images using transfer learning with pre-trained convolutional neural networks (VGG16, MobileNet, ResNet50) via Keras. MLflow is used throughout training to log hyperparameters, track metrics across experiments, store model artifacts and compare runs. The deployment pipeline uses GitHub Actions to build a Docker image on each commit, push it to AWS ECR and deploy the updated application to an EC2 instance, where a Streamlit app serves real-time predictions. Data ingestion is designed using the Strategy pattern with YAML configuration, supporting local, Kaggle and Google Drive sources interchangeably.

Data source

Kaggle chest CT scan dataset for binary cancer classification. Ingestion method is configurable via YAML and supports Kaggle API, Google Drive or local file paths without code changes.

Kaggle DatasetYAML Config
Pre-trained Models

Three Keras pre-trained CNN architectures were used and compared, all fine-tuned on the chest CT scan dataset. MLflow tracked each model's training run separately, enabling direct metric comparison across architectures.

VGG16
Deep CNN · Oxford VGG Group

16-layer deep convolutional network with uniform 3×3 filter architecture. Strong baseline for image classification with well-understood feature representations from ImageNet pretraining.

MobileNet
Lightweight CNN · Google

Depth-wise separable convolutions for efficient inference with significantly fewer parameters. Suited for resource-constrained deployment while maintaining competitive accuracy.

ResNet50
Residual Network · Microsoft

50-layer residual network with skip connections that solve the vanishing gradient problem. Enables deeper feature extraction without degradation, strong performance on medical imaging tasks.

Experiment Tracking with MLflow
01
Data Ingestion
Dataset source is configured in a YAML file and Strategy pattern allows switching between Kaggle API, Google Drive and local paths without changing code. Raw CT scan images are fetched and staged for preprocessing.
Strategy Pattern · YAML Config
02
Preprocessing & Augmentation
CT scan images are resized to match each model's input requirements, normalized and augmented with flips, rotations and zoom to improve generalization on the limited medical dataset.
Keras ImageDataGenerator
03
Transfer Learning Fine-tuning
Pre-trained ImageNet weights are loaded for each model. The classification head is replaced with task-specific dense layers and upper layers are fine-tuned on the CT scan dataset with configurable training parameters.
Keras · VGG16 · MobileNet · ResNet50
04
MLflow Experiment Logging
Each training run is logged to MLflow capturing hyperparameters, epoch-level metrics and the trained model artifact for versioning and comparison.
MLflow · Experiment Tracking · Model Registry
05
Model Comparison & Selection
MLflow UI enables side-by-side comparison of all tracked runs. The best-performing model is registered to the MLflow Model Registry for downstream deployment use.
MLflow UI · Model Registry
What MLflow tracks Per run

Each experiment run logs model architecture, learning rate, batch size, number of epochs, dropout rate, training loss, validation loss, training accuracy, validation accuracy and the serialized model artifact , enabling full reproducibility and cross-run comparison.

Key Learnings
Learned
MLflow Experiment Tracking
Managing and comparing multiple training runs, across different model architectures and hyperparameter sets quickly becomes unmanageable without structured tracking.
Takeaway
MLflow's experiment logging, metric comparison UI and model registry provide a structured workflow for reproducible ML experimentation that scales across team members and projects.
Learned
Containerised ML Deployment
Deploying ML applications to cloud infrastructure introduces dependency management and environment consistency challenges that don't exist in local development.
Takeaway
Docker containers encapsulate the full application environment, making deployments fully reproducible and eliminating environment-specific failures between development and production.
Learned
CI/CD for ML Pipelines
Manually building and deploying ML applications after each change is slow, error-prone and doesn't scale to team-based or frequent-iteration workflows.
Takeaway
GitHub Actions automates the full build-push-deploy pipeline on every commit, enforcing consistent deployment practices and making the release process immediate and reliable.
Learned
Configurable Data Ingestion
Hardcoding data sources into training pipelines creates brittle code that breaks when datasets move, team members have different access or ingestion methods need to change.
Takeaway
The Strategy pattern with YAML configuration separates the ingestion logic from the pipeline, making it trivial to switch between Kaggle, Google Drive or local data sources without touching training code.
Outcome

This project successfully delivered a working chest cancer classification system while building practical experience across the full MLOps stack, from experiment tracking with MLflow, through containerized deployment with Docker to cloud serving on AWS EC2 via an automated GitHub Actions pipeline.

Beyond the classification task itself, the project demonstrates how the infrastructure surrounding ML models like experiment logging, reproducible packaging, automated deployment and configurable pipelines is as important as model accuracy when building systems that need to be maintained, iterated on and operated in production.
